Hello World: The Back

How I built a basic static site generator, and why you should never follow in my footsteps

Web Development Typescript

I've finally decided to organise my life a little more and start writing some posts for the cool programming stuff I do in my (limited) spare time. Part of this involves updating my website to be a bit less... atrocious to manage.

Since I'm just writing some simple blog posts, a static site generator is the way to go. There's a ton of options out there: Jekyll, Hugo, and Hexo just to name a few. Of these, Jekyll is my personal favourite, but I decided to torture myself and make my own.

This post is mostly going to be boring talk about design decisions - I promise that future articles will have cool code and more interesting content!

The Basics

Static site generators are (relatively) simple to build. There are two main steps to worry about:

  • Take written posts and convert them into some HTML content
  • Take HTML content (and metadata) and stick it in a template

As with all software projects, there's a bit more complexity once you get into it, but the general process isn't too complicated. In this case, most of the code I've written is just about finding files in folders - the bulk of the work is sticking other packages together.

Infrastructure

I'm in love with Typescript lately, so that's what I'm using to build the static site generator.

There are many easy ways to host a static site - it's a lot easier to serve a folder of files than run a full backend. Github Pages or Amazon's S3 + Cloudfront would be great ways to get this working on a very tight budget.

Since I already have a VPS setup for other projects, I've just got a basic Express server hosting the generated static files. This is overkill - but it gives me a bit more room to build an actual backend if I ever need one.

Markdown

Markdown is pretty much the standard for static site generators, and rightfully so. It's easy to read, easy to write, and there are tons of tools out there for it already.

I'm writing my posts in Markdown and using Marked to convert them into HTML content, which can then be added into templates as needed. Marked works well enough, and it's easy to extend if I ever want to get a bit fancier with my layouts.

Liquid

I'm a big fan of Shopify's Liquid templating language. It lets me write templates that make sense to me as a programmer, and focus more on what I'm writing than fiddling with template files.

LiquidJS is a great Typescript implementation of Liquid, so it slots in easily with everything else so far.

Specifying default values for parameters is easy:

<h1>{{ title | default: "Cool Title" }}</h1>

Even making a for loop is simple:

{%- for post in posts %}
<section>
  <h2>{{ post.title }}</h2>
  <p>{{ post.subtext | default: "" }}</p>
</section>
{% endfor %}

These features aren't all that useful for basic article pages, but they come in very handy when doing something more complicated such as a paginated article listing.

Putting it all together

Quick recap:

  • Marked to process raw post Markdown
  • LiquidJS to put things into templates
  • Typescript to handle putting everything together
  • Prism and Sass to make the end result look nice

Overall, the result is functional and flexible. There's definitely a lot more tinkering about that I need to do, but that's mostly on the styling side of things, and I'm pretty happy with how everything has turned out so far.

The code for this static site builder is in my main Github repository. There's not too much to it, as it's mostly just combining other libraries together, but hopefully it's interesting enough.

Future Plans

I still need to do a nicer command-line interface for everything - it'd be great to link it into VSCode so that I can automatically build a page when I edit it.

A pretty common feature of other static site generators is the ability to 'include' snippets into multiple templates. This wouldn't be too difficult to add, but since I only have a handful of templates so far, this isn't all that useful.

I'd love to experiment with Pandoc to convert other formats into Markdown. If I'm writing about Python, it'd be great to take a Jupyter notebook and turn that into a post.
